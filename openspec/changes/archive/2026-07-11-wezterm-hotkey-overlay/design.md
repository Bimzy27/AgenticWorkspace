# Design: WezTerm Mux Hotkey Overlay

## Context

`.config/wezterm/wezterm.lua` defines ~25 leader-key bindings (Ctrl+Space prefix, tmux-style) directly in `config.keys`.
There is no in-terminal way to discover them.
The config is plain Lua evaluated by WezTerm at startup, so any solution must be expressible in that single file with WezTerm built-ins.

## Goals / Non-Goals

**Goals:**

- Every leader binding is visible in an overlay invoked by `LEADER ?`.
- The overlay and the real bindings are generated from one data table so they cannot drift.
- Selecting an entry in the overlay executes that binding's action.
- Zero new dependencies and no behavior change to existing bindings.

**Non-Goals:**

- Documenting non-leader bindings (copy-mode key table, WezTerm defaults like Ctrl+Shift+*).
  Copy-mode keys are listed as a static footer note only, not executable entries.
- Any external cheatsheet rendering (gum, glow, spawned panes).
- Changing any existing key assignment.

## Decisions

1. **Overlay mechanism: `act.InputSelector`** over toast notifications or a spawned cheatsheet pane.
   InputSelector is a native fuzzy-searchable overlay: instant, dismisses with Escape, needs no dependencies, and its callback can execute the chosen action via `window:perform_action`.
   A toast is too small for ~25 entries; a spawned pane adds dependencies and pane-lifecycle cleanup.

2. **Single source of truth: a `mux_bindings` list of `{ key, desc, action }` records.**
   `config.keys` is generated by iterating this table (all entries get `mods = 'LEADER'`), and the overlay choices are generated from the same table.
   Alternative considered: keeping `config.keys` handwritten and maintaining a parallel description list - rejected because the two would inevitably drift.

3. **Overlay entries carry a numeric `id` indexing back into `mux_bindings`**, and the callback performs `mux_bindings[tonumber(id)].action`.
   This keeps actions as first-class WezTerm action objects rather than re-encoding them as strings.

4. **Binding: `LEADER ?`** following the tmux `prefix ?` convention for listing keybindings.
   The `?` entry appears in the table itself so the overlay is self-documenting.

5. **Label format: fixed-width key column** (`string.format('LEADER %-4s %s', key, desc)`) so descriptions align in the monospace overlay.

## Risks / Trade-offs

- [Entries with `PromptInputLine` or `confirm = true` open a second prompt when executed from the overlay] → Acceptable: this mirrors pressing the key directly; no special-casing needed.
- [`InputSelector` visual style is fixed by WezTerm] → Acceptable: it inherits the Catppuccin scheme's overlay colors; no custom rendering is in scope.
- [Refactoring all bindings into a table risks a typo silently dropping a binding] → Mitigation: verification step diffs the generated `config.keys` against the original binding set (see tasks).

## Migration Plan

Single-file edit to `wezterm.lua`.
WezTerm reloads config automatically on save; rollback is `git checkout` of the file.
